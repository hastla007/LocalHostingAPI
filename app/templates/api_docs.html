API Overview

The Local Hosting API allows clients on your LAN to upload files and retrieve them later using simple HTTP requests.

Base URL: {{ request.url_root.rstrip('/') }}

Files expire automatically according to the retention policy configured on the settings page. Current bounds allow custom expirations between {{ config.retention_min_hours }} and {{ config.retention_max_hours }} hours with a default of {{ config.retention_hours }} hours.

Authentication

API authentication is {% if config.api_auth_enabled %}currently enabled{% else %}disabled{% endif %} by default. When authentication is enabled on the settings page, every API request must include a valid key generated from the dashboard.

  • Provide the key via the X-API-Key header (recommended) or as an Authorization: Bearer <key> header.
  • The api_key query-string parameter is also honoured for simple integrations.
  • Keys affect all upload endpoints, including the S3- and Box-compatible routes documented below.

The dashboard automatically applies the key marked as “Dashboard Default” for browser uploads. Rotate or revoke keys at any time from the settings page.

Authenticated cURL Example

curl -H "X-API-Key: YOUR_API_KEY" \
  -F "file=@/path/to/your/file" \
  {{ request.url_root.rstrip('/') }}/fileupload

Upload a File

Endpoint: POST /fileupload

Send a multipart/form-data request with one or more fields named file. Every file in the payload is stored with the same retention window supplied in the request.

Optional fields:

  • retention_hours — number of hours to keep this upload (must be within the configured range).

Single File cURL Example

curl -F "file=@/path/to/your/file" \
  {{ request.url_root.rstrip('/') }}/fileupload

Multi-file cURL Example

curl -F "file=@/path/to/first.txt" \
  -F "file=@/path/to/second.jpg" \
  -F "retention_hours=6" \
  {{ request.url_root.rstrip('/') }}/fileupload

Status Codes:

  • 201 Created – file was stored successfully.
  • 400 Bad Request – the request was invalid (missing file, bad filename, or retention outside the allowed range).

S3-Compatible Uploads

The Local Hosting API exposes lightweight Amazon S3 style endpoints for drop-in compatibility with tooling that expects object storage semantics.

POST Form Upload

Endpoint: POST /s3/<bucket>

Send a multipart form submission that includes the standard S3 fields such as key and file. Optional retention can be provided with the custom field x-amz-meta-retention-hours.

Include the X-API-Key header when API authentication is enabled.

curl -X POST \
  -F "key=uploads/${filename}" \
  -F "x-amz-meta-retention-hours=12" \
  -F "file=@/path/to/your/file.txt" \
  {{ request.url_root.rstrip('/') }}/s3/example-bucket

Successful uploads return 201 Created with an XML payload mirroring S3’s <PostResponse> including the resolved object location, key, and ETag. The x-localhosting-file-id header exposes the internal identifier so you can still reach the classic download endpoints if needed.

PUT Object Upload

Endpoint: PUT /s3/<bucket>/<key>

Issue a raw PUT with the file contents in the request body. Custom retention is accepted through the X-Amz-Meta-Retention-Hours header or a retentionHours query parameter.

Provide the same X-API-Key header if authentication is enabled for the service.

curl -X PUT \
  -H "Content-Type: video/mp4" \
  -H "X-Amz-Meta-Retention-Hours: 24" \
  --data-binary @movie.mp4 \
  {{ request.url_root.rstrip('/') }}/s3/example-bucket/media/movie.mp4

A successful PUT returns 200 OK with an S3-style <PutObjectResult> XML body alongside the generated ETag and object location headers.

Box-Compatible Endpoints

To integrate with SDKs or automation that expect Box.com’s file APIs, Local Hosting API exposes lightweight compatibility routes that mirror the core upload and download behaviour.

Upload Content

Endpoint: POST /2.0/files/content

Send a multipart form-data request with a file field named file. Optional metadata can be supplied via an attributes field containing JSON, matching the payload format the Box API expects. The response returns an entries array with Box-style metadata including SHA-1 hashes and shareable download URLs.

All Box-compatible requests accept the X-API-Key or Authorization header when API authentication is active.

curl -X POST \
  -F "attributes={\"name\":\"report.pdf\"}" \
  -F "file=@/path/to/report.pdf" \
  {{ request.url_root.rstrip('/') }}/2.0/files/content

Sample JSON Response

{
  "total_count": 1,
  "entries": [
    {
      "type": "file",
      "id": "1234567890",
      "sequence_id": "0",
      "etag": "0",
      "sha1": "6a9c1b5b6f5b88f16a5aa764f4144f8aa5a61234",
      "name": "report.pdf",
      "size": 523980,
      "created_at": "2024-06-04T18:43:07Z",
      "modified_at": "2024-06-04T18:43:07Z",
      "expires_at": "2024-06-07T18:43:07Z",
      "retention_hours": {{ config.retention_hours }},
      "download_url": "{{ request.url_root.rstrip('/') }}/2.0/files/1234567890/content",
      "direct_download_url": "{{ request.url_root.rstrip('/') }}/files/1234567890/report.pdf",
      "raw_download_url": "{{ request.url_root.rstrip('/') }}/report.pdf"
    }
  ]
}

Download Content

Endpoint: GET /2.0/files/<id>/content

Streams the stored file back with the original filename and content type. The returned payload is the raw file bytes, so you can pipe the response directly to disk. 

curl -L -o report.pdf {{ request.url_root.rstrip('/') }}/2.0/files/<id>/content

File Request Metadata

Endpoint: GET /2.0/file_requests/<id>

Returns a JSON document describing the stored file with fields such as title, status, upload_url, and url so Box automations can link directly to the compatible upload and download routes.

Sample JSON Response

{
  "type": "file_request",
  "id": "1234567890",
  "title": "Upload report.pdf",
  "description": "Local Hosting API Box-compatible upload",
  "status": "active",
  "retention_hours": {{ config.retention_hours }},
  "upload_url": "{{ request.url_root.rstrip('/') }}/2.0/files/content",
  "url": "{{ request.url_root.rstrip('/') }}/2.0/files/1234567890/content",
  "download_url": "{{ request.url_root.rstrip('/') }}/download/1234567890",
  "direct_download_url": "{{ request.url_root.rstrip('/') }}/files/1234567890/report.pdf",
  "raw_download_url": "{{ request.url_root.rstrip('/') }}/report.pdf"
}

Download a File

Endpoint: GET /download/<id>

Use the download_url returned during upload or substitute the file ID:

curl -O {{ request.url_root.rstrip('/') }}/download/<id>

A second direct endpoint preserves the original filename in the URL for easier sharing: GET /files/<id>/<original_filename>. 

curl -O {{ request.url_root.rstrip('/') }}/files/<id>/document.pdf

New uploads also receive a raw filename URL that omits the ID entirely, ideal for inline playback or embedding: GET /<original_filename>. If a filename is already in use, a numeric suffix (for example report-1.pdf) is appended to keep the paths unique. 

curl -O {{ request.url_root.rstrip('/') }}/document.pdf

All endpoints honor the original filename and require the file to be unexpired; otherwise a 404 Not Found is returned.

Managing Stored Files

Use the Hosting dashboard to view, download, or delete uploads through the web interface. To upload from a browser, visit Upload File for the drag-and-drop workflow.

Monitoring & Logs

The Logs page streams the latest application output, including upload and download lifecycle entries. Use the manual refresh button or enable one of the automatic polling intervals to keep the view up to date. If you expose a Docker log file via the LOCALHOSTING_DOCKER_LOG_PATH environment variable, it will appear alongside the built-in application log.
